# Опис RL-модуля та проведених експериментів

## 1. Загальна ідея

У проєкті спочатку була побудована система оцінювання транспортної доступності соціальної інфраструктури міста. Вона рахує, скільки людей може дістатися до лікарень і шкіл пішки або громадським транспортом у межах заданих часових порогів.

RL-модуль додано як інтелектуальний шар поверх цієї оцінки. Його задача не просто показати поточну доступність, а автоматично знайти, які зміни інтенсивності маршрутів можуть покращити доступність закладів.

Поточна ціль RL-модуля:

```text
перерозподіляти транспортну інтенсивність між маршрутами так,
щоб покращити I*_peak для пріоритетних закладів і не погіршити
систему безконтрольно.
```

## 2. Які дані використовуються

Основні вхідні файли:

```text
data/processed/catchment_buildings_baseline.parquet
data/processed/accessibility_index_baseline.csv
data/processed/facility_entropy_baseline.parquet
data/processed/building_weights_baseline.parquet
gtfs_static/easyway_routes.csv
gtfs_static/easyway_metro.csv
gtfs_static/osm_easyway_data.csv
gtfs_static/osm_easyway_metro_data.csv
```

`easyway_routes.csv` / `easyway_metro.csv` використовуються як fallback-джерело розкладів і модельної інтенсивності.

`pipeline/transports/bus/*.csv` та `pipeline/transports/trol/*.csv` використовуються як диспетчерські розклади для автобусів і тролейбусів, коли такі файли доступні.

`osm_easyway_data.csv` використовується як фільтр валідності маршрутів. Якщо в `config.toml` задано:

```toml
require_osm_mapping = true
```

то система бере тільки ті маршрути, які є в OSM-зв'язці. Це потрібно, щоб не включати приміські або непідтверджені маршрути, наприклад `324P`.

## 3. Як рахується початкова частота маршруту

Для автобусів і тролейбусів, якщо є диспетчерський CSV, початкова фізична інтенсивність рахується так:

```text
current_freq_per_hour = peak_trips / total_peak_hours
```

де `peak_trips` це кількість повних рейсів маршруту в пікові вікна `[07:00, 09:00)` та `[17:00, 19:00)`, а `total_peak_hours = 4`.

Для метро або маршрутів без диспетчерського CSV використовується fallback з EasyWay:

```text
current_freq_per_hour = easyway_peak_departures_from_first_stops / total_peak_hours
```

Тобто беруться пікові відправлення з першої зупинки кожного напрямку, а не сума часових записів по всіх зупинках. Стару логіку `total_departures / 11` прибрано, бо вона множила один рейс на кількість зупинок і завищувала інтенсивність.

Далі частота переводиться в RL-шкалу `1..12`. Це не реальні рейси на годину, а нормалізована інтенсивність для алгоритму.

Нормалізація виконується окремо для кожного типу транспорту:

```text
bus окремо
trol окремо
tram окремо
metro окремо
```

Це потрібно, бо автобуси, тролейбуси, трамваї і метро мають різну фізичну природу частот.

У `config.toml` можна вибрати спосіб масштабування:

```toml
freq_scaling = "log"
```

або:

```toml
freq_scaling = "linear"
```

`log` стискає великий розкид частот і дає кращу роздільну здатність для середніх маршрутів.

`linear` залишає абсолютні співвідношення частот без логарифма.

## 4. Який action space використовується

Спочатку тестувався простий action space:

```text
маршрут +1
маршрут -1
```

Потім від цієї логіки відмовилися, бо вона дозволяла просто збільшувати або зменшувати маршрут без явного балансу ресурсу.

Поточна логіка:

```text
donor route - action_step
receiver route + action_step
```

Тобто один маршрут виступає донором ресурсу, інший отримує цей ресурс.

Приклад:

```text
bus 43K: -0.5
bus 43:  +0.5
```

Дії дозволені тільки в межах одного типу транспорту:

```text
bus -> bus
trol -> trol
tram -> tram
metro -> metro
```

Це найбільш обережна постановка, бо різні типи транспорту мають різні бюджети, місткість і експлуатаційні обмеження.

Також додано сценарний режим:

```toml
allow_cross_type_transfers = true
```

У цьому режимі дозволені переноси між різними типами транспорту:

```text
metro -> bus
metro -> trol
bus -> tram
```

Такий режим треба трактувати як what-if сценарій, а не як прямий бюджетний план. Одна одиниця RL-шкали для метро і одна одиниця RL-шкали для автобуса не є фізично однаковими ресурсами.

## 5. Обмеження на зміну маршрутів

У `config.toml` є два ключові параметри:

```toml
action_step = 0.5
max_route_delta = 5
```

`action_step` визначає розмір одного кроку перерозподілу.

`max_route_delta` обмежує, наскільки сильно один маршрут може відійти від початкового стану.

Наприклад:

```text
initial_freq = 7.0
max_route_delta = 5
```

Тоді маршрут може змінюватися в межах:

```text
2.0 .. 12.0
```

Додатково маршрут не може впасти нижче `1.0` і не може перевищити `12.0`.

## 6. Як перераховується доступність після зміни частоти

Для кожного запису `building -> facility` система дивиться, який маршрут був використаний у baseline.

Якщо маршрут не змінився:

```text
new_total_min = old_total_min
```

Якщо маршрут змінився:

```text
new_wait = old_wait * (initial_freq / current_freq)
```

Потім:

```text
new_total_min = walk_in + new_wait + transit + walk_out
```

Тобто модель змінює тільки компонент очікування транспорту. Час пішки і час у транспорті лишаються такими, як у baseline.

Це свідоме спрощення:

```text
зміна частоти маршруту впливає на очікування,
але не перебудовує геометрію маршруту і не змінює час руху між зупинками.
```

## 7. Формула I*_peak

Для закладу доступність перераховується через ваги будинків:

```text
I_peak = [sum(w_b * exp(-0.05 * new_total_min)) / sum(w_city)] * Hnorm_peak
```

де:

```text
w_b          вага будинку
new_total_min новий час доступу
sum(w_city) сума ваг по місту
Hnorm_peak   ентропійний коефіцієнт закладу
```

## 8. Reward і objective

Для target-групи основний сигнал:

```text
delta_target = I_peak_after_targets - I_peak_before_targets
```

Також враховується шкода для інших закладів:

```text
non_target_harm = сума погіршень по нецільових закладах
```

У спрощеному вигляді objective:

```text
objective = delta_target - non_target_harm_weight * non_target_harm
```

Для greedy baseline додано hard-constraint:

```toml
enforce_target_non_worsening = true
target_harm_tolerance = 0.00000001
```

Якщо дія покращує середнє по target-групі, але погіршує хоча б один target-заклад більше ніж на `target_harm_tolerance`, така дія не приймається.

Для PPO додано масштабування reward:

```toml
reward_scale = 1000.0
```

Це не змінює логіку objective, але робить навчальний сигнал більшим. Без цього зміни `I*_peak` часто були на рівні `1e-6`, і PPO не мав стабільного сигналу для навчання.

Додатково можна враховувати зменшення очікування для target-закладів:

```toml
target_wait_reward_weight = 0.0
```

Якщо значення більше нуля, модель отримує додатковий сигнал за зменшення `wait`.

## 9. Що пробували

### 9.1 Один заклад H327

Перший тест був зроблений для одного закладу:

```text
H327, Здоров'я родини
```

Було реалізовано:

```text
локальна підмережа маршрутів
локальний reward тільки по H327
локальний recompute I*_peak(H327)
заборона active=False
best-state tracking
валідація відстані маршрутів до закладу
```

Результат показав, що середовище працює, але для H327 агент майже не мав простору для покращення, бо ключові маршрути вже були на високих частотах.

### 9.2 Група з кількох закладів

Далі тестувалися групи з кількох закладів:

```text
слабкі заклади
середні заклади
сильні заклади
комбіновані групи
```

Це дало більше маршрутів і більший action space, але PPO все одно часто не знаходив корисної політики.

### 9.3 Перехід від PPO до action probe

Було додано `10f_action_probe`, який перебирає всі одно-крокові дії:

```text
donor -> receiver
```

і напряму рахує, чи є позитивний ефект.

Це показало, що позитивні дії в середовищі існують. Отже проблема була не в тому, що модель доступності не реагує, а в тому, що PPO погано вловлює дуже малий і розріджений reward.

### 9.4 Greedy baseline

Після цього було додано `10g_apply_best_probe`.

Цей крок не навчає PPO, а багато разів застосовує найкращу поточну дію:

```text
1. перевірити всі donor->receiver дії
2. взяти найкращу позитивну
3. застосувати її
4. повторити, поки є покращення
```

Це стало основним практичним baseline-рішенням, бо воно стабільно знаходить корисні зміни там, де PPO часто лишається в початковому стані.

### 9.5 Практичний звіт

Було додано `10h_recommendations`.

Його задача:

```text
перекласти RL-шкалу у зрозумілі транспортні рекомендації
```

Наприклад:

```text
додати приблизно N рейсів у пік
зменшити інтервал з X хв до Y хв
маршрут може бути донором ресурсу
орієнтовна зміна рухомого складу
```

## 10. Чому PPO не став фінальним основним результатом

PPO був реалізований як прототип RL-середовища:

```text
стан = граф маршрутів
дія = перерозподіл частоти між маршрутами
reward = зміна доступності
```

Але на практиці reward виявився дуже малим:

```text
зміни I*_peak часто на рівні 1e-6 або менше
```

Для PPO це слабкий і розріджений сигнал. У багатьох експериментах найкращий стан лишався початковим.

Після цього було додано:

```text
reward_scale
target_wait_reward_weight
best-state tracking
```

Ці зміни покращують навчальний сигнал, але не гарантують стабільного покращення PPO для всіх конфігурацій.

Тому фінальна практична частина використовує:

```text
PPO як демонстрацію RL-середовища
10f/10g як контрольний і стабільний heuristic baseline
10h як практичний інтерпретатор результатів
```

## 12. Як правильно описувати результат у дипломі

Коректне формулювання:

```text
Розроблений RL-модуль дозволяє автоматично аналізувати,
які маршрути мають найбільший вплив на доступність вибраних
закладів, і формує сценарії перерозподілу транспортної
інтенсивності.
```

Важливо не стверджувати, що це готовий диспетчерський план.

Краще писати:

```text
отримані рекомендації є сценарною оцінкою;
вони показують потенційний напрямок зміни частот;
перед практичним впровадженням потрібна транспортна валідація.
```

## 13. Поточні обмеження

Основні обмеження:

```text
модель змінює тільки wait, а не час руху маршруту;
sigma очікування не перебудовується повністю;
дії дозволені тільки в межах одного типу транспорту;
місткість транспорту не моделюється;
реальний парк рухомого складу оцінюється наближено;
результат залежить від якості розкладів easyway/osm mapping;
PPO має слабкий reward і не завжди знаходить покращення.
```

Ці обмеження треба явно зазначити в дипломі.

## 14. Подальші покращення

Можливі напрямки розвитку:

```text
add-only сценарій, де рейси тільки додаються;
cross-type transfer між bus/trol/tram/metro;
окремий no-harm режим з жорстким обмеженням погіршень;
перерахунок не тільки wait, а й надійності інтервалів;
врахування місткості транспорту;
RL або heuristic для всіх закладів міста;
порівняння log і linear нормалізації частот.
```
